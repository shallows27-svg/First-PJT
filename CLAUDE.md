## Architecture

### Auth + session flow (Supabase SSR)

Three Supabase clients, never interchangeable:

1. `src/lib/supabase/client.ts` — `createBrowserClient`, for Client Components.
2. `src/lib/supabase/server.ts` — `createServerClient` with `await cookies()`. For Server Components, Server Actions, and Route Handlers. Its `setAll` swallows errors because Server Components cannot write cookies — session refresh is the proxy's job.
3. `src/lib/supabase/proxy.ts` — `createServerClient` wired to `NextRequest`/`NextResponse`. Called from `src/proxy.ts` on every non-static request to refresh tokens **and** perform an optimistic redirect to `/login` for unauthenticated hits to `/dashboard*`.

Important constraint from the proxy: do not insert any logic between `createServerClient(...)` and `supabase.auth.getUser()` — comment in the file explicitly warns "세션 꼬임 방지" (avoid session corruption).

Auth Server Actions live in `src/app/auth/actions.ts` (`signIn`, `signUp`, `signOut`). They use the `useActionState` shape: `(prevState, formData) => Promise<AuthState>`. On success they `revalidatePath("/", "layout")` then `redirect()`. The email confirm callback is a Route Handler at `src/app/auth/confirm/route.ts` that handles both `token_hash`+`type` and `code` flows.

The proxy's redirect is **optimistic only** — every protected page (e.g. `src/app/dashboard/page.tsx`) must still call `supabase.auth.getUser()` and `redirect("/login")` itself before reading data.

### Data + AI flow

Two Supabase tables (RLS-protected, one row per user, upserted on `user_id`):

- `messages(user_id, content, updated_at)` — the "오늘의 한 줄" feature.
- `portfolios(user_id, holdings, holdings_items, ai_summary, updated_at)` — auto-allocation portfolio.
  - `holdings` (text, **nullable**) — legacy free-text column, no longer written by new code.
  - `holdings_items` (jsonb, default `'[]'`) — array of `{ ticker, name, quantity, current_price, value_krw, region }`. Region is `"KR" | "GL/US" | "Cash"`.
  - `ai_summary` (text) — 3-line summary generated by `summarizePortfolioFromItems`.

`src/app/dashboard/actions.ts` exposes:
- `saveMessage` — "오늘의 한 줄" upsert.
- `analyzeScreenshots(files)` — calls Gemini Vision (`src/lib/ai/vision.ts`) per screenshot, returns merged `HoldingItem[]`. Rate-limited via `src/lib/portfolio/rate-limit.ts` (20/hour, Vercel Runtime Cache with in-memory dev fallback).
- `savePortfolio(items)` — Zod re-validates, writes `holdings_items`, regenerates AI summary, 10s debounce.
- `regenerateSummary()` — manual summary refresh.
- `refreshQuotes(tickers)` — Yahoo Finance live prices (`src/lib/quotes/yahoo.ts`), 5-min cache.

All actions re-verify `auth.getUser()` inside the action (Server Actions can be called as bare POSTs, so middleware/proxy checks are not sufficient).

`src/lib/openrouter.ts` exposes `callChatCompletion`, a thin adapter around OpenRouter's OpenAI-compatible endpoint with API key guard, AbortController timeout, and sanitized errors. **This module is server-only — never import it from a `"use client"` file.** Errors are logged server-side; user-facing messages never expose the API key or request body, and thrown errors include only HTTP status codes.

## Supabase MCP

When using Supabase MCP tools against the live project:

- Run `list_tables` first to confirm `messages` / `portfolios` schemas before any migration.
- Use `get_logs` / `get_advisors` to diagnose before applying changes.
- `apply_migration` hits the remote project directly — prefer local Supabase CLI workflows if available.
- For client-side config snippets, use `get_project_url` and `get_publishable_keys` (not the service role).
- The `holdings` NOT NULL constraint was dropped via MCP `apply_migration` during auto-allocation development; that migration is not in git as a file yet.
